1. Описание задачи "от заказчика"
Нам необходим модуль-симулятор биржи на уровне ОС. Это должен быть драйвер для Linux, написанный на языке C, который генерирует непрерывный поток данных о котировках акций. Этот драйвер выступает первоисточником данных (mock-биржей) для всей нашей торговой экосистемы. Он должен уметь имитировать изменение цен на различные тикеры в реальном времени, обеспечивая высокую пропускную способность, чтобы наш Go-микросервис мог бесперебойно считывать эти данные для дальнейшей трансляции клиентам (с расчетом на нагрузку до 10 000 клиентов). Драйвер должен работать стабильно, не вызывая сбоев в ядре (kernel panics), и предоставлять удобный интерфейс для чтения потока котировок.




2. Техническая реализация
Драйвер будет представлять из себя Блочное устройство через sysfs. Решение обосновано необходимостью большой пропускной способности и одновременной выгрузки больших массивов данных. 
Математика изменения котировок будет реализована с помощью алгоритма шума Перлина. Такой подход позволит не производить вычисления для генерации случайных значений при каждом запросе к драйверу, а лишь передвигаться по заранее сгенерированной маске шума, равновероятно выбирая направление движения. Разные котировки будут начинать свой путь из разных точек маски. 
Так как UNIX по умолчанию поддерживает одновременное чтение из файлов, подобный подход позволит ограничить количество клиентов ресурсами системы, узкого горлышка в месте драйвера не возникнет.

Достаточно подробно описал на схеме, вот текстовое описание:
Контракт с Go является единственным producer исторических котировок для ClickHouse. На входе он получает из C-драйвера структуры вида struct mock { char ticker[8]; float price; }, после чего обязан нормализовать событие: преобразовать ticker в строковый symbol, зафиксировать event_time, привести цену к корректному типу и добавить ingested_at. Таким образом, перед записью в ClickHouse формируется доменная запись временного ряда. Запись выполняется через библиотеку clickhouse-go по native TCP protocol. Используются порты: 9000 — native TCP; 8123 — HTTP interface. Вставка выполняется только батчами: до 10000 записей или flush каждые 200ms. Одиночные insert’ы запрещены. При graceful shutdown остаток буфера должен быть дописан, а при временных ошибках используется retry с backoff. 

Контракт с Ktor является единственным backend-потребителем исторических данных ClickHouse. Мобильные клиенты не обращаются к ClickHouse напрямую. Ktor использует ClickHouse для чтения: истории цены за период; свечей по интервалу; агрегатов min/max/first/last; последней исторической цены. Типовые запросы Ktor: line chart — список точек (time, price); candlestick chart — список свечей (time, open, high, low, close); summary — агрегаты по диапазону. Таким образом, ClickHouse возвращает Ktor не “графики”, а подготовленные данные для их построения на клиенте.

Mini-SRS: Модуль брокера сообщений и быстрого кэша (Уровень Service)
Команда модуля:
* Тимлид: ttemmaaa
* Саппорт (подчиненный): Pyhardcoding
1. Описание задачи "от заказчика"
Нам необходим высокоскоростной брокер сообщений и in-memory кэш на базе Redis (или KeyDB). Этот модуль решает две критические задачи для обеспечения работы терминала на 10 000 клиентов. Во-первых, он принимает горячий поток изменения цен (котировок) от Go-сервиса и моментально транслирует его в Ktor-бэкенд для обновления графиков и цен на устройствах пользователей в реальном времени. Во-вторых, он может использоваться как кэш для снижения нагрузки на основную транзакционную базу (PostgreSQL), например, для хранения текущих сессий пользователей или "стаканов" заявок. Решение должно быть надежным, легко разворачиваемым и отдавать метрики для системы Телеметрии.
2. Техническая реализация
(Опиши, как будет настроен инстанс. Будет ли это классический Redis или KeyDB (как упоминалось в лекции)? Какой механизм рассылки котировок будет использоваться: классический Pub/Sub или Redis Streams для гарантии доставки? Будет ли включена персистентность (RDB/AOF), или модуль будет работать исключительно в памяти? Как будут настроены лимиты памяти (maxmemory-policy)?)
В качестве основной технологии используется обычный Redis 7.x, а не KeyDB. Причина простая: Redis лучше знаком команде, у него понятнее экосистема, больше готовых примеров и меньше риск неожиданных проблем на этапе интеграции. Для учебного проекта это более надежный и предсказуемый выбор.
Для межсервисной передачи котировок в рамках проекта фиксируется единый транспорт — Redis Streams. Выбор сделан в пользу Streams, поскольку этот механизм позволяет не просто отправить событие, а сохранить его на короткое время, дочитать после временного сбоя и контролировать подтверждение обработки. Таким образом, базовый интеграционный контракт между сервисами строится именно на Streams.
* Основной поток котировок идет через stream stream:quotes:v1.
* Ktor читает этот поток через consumer group cg:ktor:quotes.
Redis используется не только как брокер сообщений, но и как быстрый in-memory кэш. В кэше предлагается хранить горячие данные, к которым Ktor обращается особенно часто: активные сессии пользователей, последние цены по инструментам и краткоживущие снимки стаканов заявок.
По надежности предлагается включить AOF в режиме appendfsync everysec. Такой вариант дает разумный компромисс: сервис не работает полностью “в пустой памяти”, но и не становится слишком тяжелым по диску. Дополнительно можно оставить RDB-снапшоты как резервный механизм восстановления, но главным способом сохранения состояния будет именно AOF.
* Режим хранения: AOF everysec + периодические RDB snapshots.
* Авторизация: пароль через переменные окружения, без хардкода в репозитории.
* Развертывание: отдельный контейнер Redis + отдельный контейнер redis-exporter в docker-compose.
* Проверка живости: healthcheck через redis-cli ping.
Так как Redis одновременно используется и как кэш, и как брокер, важно заранее ограничить память и поведение при переполнении. Предлагается задать maxmemory на уровне стенда и использовать политику volatile-ttl. Это значит, что при нехватке памяти Redis будет в первую очередь вытеснять временные кэш-ключи с TTL, а не системные данные и не важные служебные структуры. Для Streams дополнительно включается ограничение длины через MAXLEN ~, чтобы поток котировок не рос бесконечно.
* Рекомендуемый maxmemory: задается через конфиг стенда, например 1–2 ГБ для dev/test.
* Политика вытеснения: maxmemory-policy volatile-ttl.
* Ограничение стрима: XADD ... MAXLEN ~ 100000.
Итоговая идея реализации такая: Redis остается легким и быстрым сервисом, который закрывает две роли сразу — передает горячие данные между модулями и снимает часть нагрузки с PostgreSQL за счет кэширования. При этом архитектура остается достаточно простой, чтобы ее реально поднять, проверить и защитить на проекте.

3. Контракты взаимодействия (Data Contracts)
(Опиши стыки с тремя соседними модулями:
1. С модулем Go: В какие топики/каналы Go должен пушить данные? Какой формат сериализации (JSON, Protobuf)?
Go-сервис отвечает за генерацию и отправку горячего потока котировок. Redis со своей стороны не валидирует бизнес-смысл цены, а принимает уже подготовленное сообщение в согласованном формате. Поэтому ответственность здесь делится так: Go формирует корректное событие, Redis гарантирует его быструю доставку и короткое хранение, а Ktor затем это событие дочитывает.
Зафиксированный контракт публикации:
* Поток публикации: stream:quotes:v1.
* Команда записи: XADD stream:quotes:v1 MAXLEN ~ 100000 * payload <protobuf-bytes>.
* Формат payload: бинарный Protobuf.
* Redis хранит payload как есть, без бизнес-валидации и без интерпретации внутренних полей.
В качестве формата сериализации выбирается Protobuf, а не JSON. Это решение принято для того, чтобы у всех сервисов был единый строгий контракт, не зависящий от ручной интерпретации полей. По сравнению с JSON Protobuf дает более компактное представление сообщения, лучше подходит для высокочастотного обмена и удобнее для согласованной интеграции между Go и Ktor.
Каноническая схема сообщения определяется через .proto-контракт команды. Для первой версии проекта используется схема уровня quotes.v1. Пример структуры сообщения:  
Правила версионирования:
* текущая версия контракта: v1;
* при несовместимом изменении схемы создается новая версия контракта;
* изменение версии должно быть одинаково отражено у всех участников интеграции.
Таким образом:
* Go формирует корректное сообщение по protobuf-схеме;
* Redis Streams принимает и хранит бинарный payload без бизнес-валидации;
* Ktor считывает и десериализует сообщение по той же protobuf-схеме.






2. С модулем Ktor: Как Ktor подписывается на эти данные? Какие ключи (keyspace) выделены под кэширование запросов Ktor?
Ktor использует Redis в двух ролях: во-первых, как источник горячих событий для real-time обновлений, а во-вторых, как быстрый кэш перед обращением в PostgreSQL. Чтобы не было хаоса по ключам, keyspace нужно зафиксировать заранее.
• Consumer group для чтения котировок: `cg:ktor:quotes`.
• Имя consumer-а: `ktor-instance-{n}`.
• После успешной обработки события Ktor обязан отправить `XACK`.
• При временном сбое экземпляр Ktor может дочитать pending-сообщения, не теряя данные.
Предлагаемая структура ключей для кэша:
Ключ
	Назначение
	TTL
	Комментарий
	`cache:session:{sessionId}`
	Сессия пользователя
	30 мин
	Используется для быстрой проверки активной сессии.
	`cache:price:{symbol}`
	Последняя цена инструмента
	5 сек
	Нужен для быстрого ответа без похода в БД.
	`cache:orderbook:{symbol}`

	Очень горячие и короткоживущие данные.

	Опциональный сервисный кэш.
	Отдельно важно зафиксировать правило: все временные кэш-ключи обязаны иметь TTL. Это защищает систему от незаметного разрастания памяти и помогает Redis оставаться именно быстрым слоем, а не второй «вечной» базой данных.
3. С модулем Telemetry: Как снимаются метрики с самого Redis (например, через redis-exporter для Prometheus/OpenTelemetry)?)
С точки зрения наблюдаемости Redis не должен быть «черным ящиком». Команда Telemetry должна видеть, жив ли инстанс, сколько памяти он ест, не начал ли он вытеснять ключи и не уперся ли в лимит по операциям.
• Инфраструктурные метрики снимаются через `redis-exporter`.
• Exporter поднимается отдельным контейнером и отдает метрики на свой порт, например `9121`.
• Далее эти метрики забираются системой Telemetry через Prometheus/OpenTelemetry-совместимый пайплайн.
Минимальный набор метрик, который должен быть доступен:
• `redis_up` — доступность Redis.
• `redis_memory_used_bytes` — фактическое потребление памяти.
• `redis_connected_clients` — число подключенных клиентов.
• `redis_commands_processed_total` и `redis_instantaneous_ops_per_sec` — нагрузка по командам.
• `redis_keyspace_hits_total` / `redis_keyspace_misses_total` — полезность кэша.
• `redis_evicted_keys_total` — признаки нехватки памяти.
• `redis_rejected_connections_total` — отказ в подключениях.

2. Техническая реализация
(Опиши архитектуру базы. Как спроектирована ER-модель (какие основные таблицы: Users, Accounts, Orders, Portfolios)? Какие индексы планируешь использовать для ускорения частых выборок (например, истории заявок юзера)? Как будет обеспечиваться ссылочная целостность (Foreign Keys) и консистентность финансов (транзакции, уровни изоляции, блокировки FOR UPDATE)? Как разворачиваем: Docker, docker-compose?)
* PostgreSQL хранит только транзакционные данные: пользователей, счета, позиции портфеля, заявки и финансовые операции. Исторические котировки не пишем сюда, они принадлежат ClickHouse; текущие котировки приходят в Ktor через Redis и используются только для валидации цены сделки.
* Базовая ER-модель:
   * users(id uuid primary key, email citext unique not null, password_hash text not null, full_name text, role text not null default 'CLIENT', created_at timestamptz not null default now(), updated_at timestamptz not null default now());
   * accounts(id uuid primary key, user_id uuid not null references users(id), currency char(3) not null default 'RUB', cash_balance numeric(20, 4) not null check (cash_balance >= 0), reserved_balance numeric(20, 4) not null default 0 check (reserved_balance >= 0), created_at timestamptz not null default now(), unique(user_id, currency));
   * instruments(ticker varchar(12) primary key, name text not null, currency char(3) not null default 'RUB', lot_size int not null default 1 check (lot_size > 0), is_active boolean not null default true, last_price numeric(20, 4));
   * (поле last_price обновляется Ktor при получении каждого тика из Redis и используется как fallback при перезапуске сервиса — до прихода первого тика из Redis берётся именно оно; если last_price IS NULL — заявка отклоняется с QUOTE_UNAVAILABLE);

* Денежные значения храним в numeric(20, 4), количество бумаг в numeric(20, 8). float/double для денег запрещены.

* Финансовые операции выполняются в транзакции PostgreSQL. Для покупки Ktor блокирует строку счета через SELECT ... 3. Контракты взаимодействия (Data Contracts)
(Как именно Ktor-сервис будет общаться с базой?)
1. Схема и миграции: Как Ktor будет накатывать структуру (используем Flyway, Liquibase или голые SQL-скрипты при старте)?
2. Доступ к данным: Ktor будет использовать голый SQL (как рекомендовано в лекции) или легковесные обертки (Exposed)? Как будут передаваться параметры подключения (URL, логин, пароль, пул соединений)?
Схема и миграции: источник истины по структуре БД - Flyway-миграции в Ktor-репозитории: db/migration/V001__init.sql, V002__indexes.sql, далее только forward-only миграции. Ktor запускает Flyway на старте приложения до поднятия HTTP-роутинга. Ручное изменение схемы в контейнере запрещено.
Подключение: Ktor подключается по JDBC через HikariCP. Переменные окружения: POSTGRES_HOST=postgres, POSTGRES_PORT=5432, POSTGRES_DB=broker, POSTGRES_USER=broker_app, POSTGRES_PASSWORD=..., POSTGRES_MAX_POOL_SIZE=30, POSTGRES_CONNECTION_TIMEOUT_MS=3000.
Доступ к данным: используем обычный SQL через JdbcTemplate-подобный DAO/репозитории или Jdbi; тяжелые ORM и автогенерацию схемы через Exposed не используем, чтобы транзакции и блокировки были явно видны в коде. Все параметры передаются только через prepared statements.
Права: приложение работает под ролью broker_app с правами select/insert/update/delete только на прикладные таблицы. Отдельная роль broker_migration используется Flyway для DDL. Пароль и URL не коммитятся, передаются через .env/Docker secrets.
Контракт транзакции покупки: Ktor открывает DB transaction, читает последнюю цену из своего Redis-кэша (если цены нет — возвращает QUOTE_UNAVAILABLE), затем всегда в фиксированном порядке: (1) SELECT * FROM accounts WHERE user_id = :userId AND currency = :currency FOR UPDATE, (2) SELECT * FROM portfolio_positions WHERE user_id = :userId AND ticker = :ticker FOR UPDATE (строка может не существовать — это нормально для первой покупки), проверяет cash_balance, вставляет orders(status='FILLED'), делает UPDATE accounts SET cash_balance = cash_balance - :total, выполняет INSERT ... ON CONFLICT(user_id, ticker) DO UPDATE для portfolio_positions, вставляет запись в transactions(type='BUY'), коммитит. При любой ошибке транзакция откатывается, наружу возвращается API-ошибка без частичного изменения баланса.
Контракт транзакции продажи: Ktor открывает DB transaction, затем в том же фиксированном порядке что и BUY: (1) SELECT * FROM accounts WHERE user_id = :userId AND currency = :currency FOR UPDATE, (2) SELECT * FROM portfolio_positions WHERE user_id = :userId AND ticker = :ticker FOR UPDATE, проверяет quantity >= :requested, уменьшает позицию (UPDATE portfolio_positions SET quantity = quantity - :requested), если quantity становится 0 — avg_price обнуляется (avg_price = 0) и строка остается для аудита, увеличивает cash_balance, вставляет orders(status='FILLED') и transactions(type='SELL'), коммитит. Важно: порядок блокировок accounts → portfolio_positions обязателен в обоих сценариях для исключения deadlock при конкурентных запросах одного пользователя.
Частичное исполнение: в v1 не поддерживается. Заявка исполняется целиком (FILLED) или отклоняется (REJECTED). Статус PARTIAL зарезервирован для v2.
Границы ответственности: PostgreSQL не подписывается на Redis и не знает о WebSocket. Единственный потребитель БД - Ktor. Go пишет историю в ClickHouse, но не пишет в PostgreSQL напрямую.

Сервис напрямую работает с PostgreSQL для надежного сохранения транзакций и состояния пользователей. Одновременно с этим Ktor должен получать «горячие» котировки из Redis и транслировать их на мобильные устройства в реальном времени. Система должна быть спроектирована асинхронно (на корутинах) для удержания до 10 000 одновременных подключений. 2. Техническая реализация
(Опиши архитектуру приложения. Как будет реализована асинхронность (корутины)? Как Ktor будет отдавать котировки в реальном времени (WebSockets или Server-Sent Events - SSE)? Какую библиотеку используем для сериализации (kotlinx.serialization)? Как реализована авторизация (JWT, сессии)? Как сервис будет обрабатывать ошибки и валидировать входящие данные от клиентов?)
* Ktor-приложение строится слоями routing -> service -> repository -> database/redis. Роуты не содержат SQL и финансовой логики; они валидируют DTO, достают userId из JWT и вызывают сервисы.
* Асинхронность: Ktor Netty + Kotlin coroutines. Блокирующие JDBC-запросы выполняются на отдельном Dispatchers.IO/database dispatcher, чтобы не блокировать event loop. Redis-подписка работает в отдельной coroutine scope с graceful shutdown.
* Сериализация: kotlinx.serialization, формат JSON, поля в REST/WebSocket - camelCase. Время передается как ISO-8601 UTC строка (2026-04-11T12:30:00Z) или timestampMs для котировок.
* Авторизация: JWT access token в заголовке Authorization: Bearer <token>, refresh token хранится сервером в PostgreSQL в виде хэша. Пароли хэшируются Argon2id/BCrypt.
* Реальное время: для мобильных клиентов используем WebSocket GET /api/v1/quotes/ws. Ktor подписан на Redis Pub/Sub канал quotes.ticks, фильтрует сообщения по тикерам подписки клиента и рассылает активным WebSocket-сессиям. Стратегия reconnect к Redis: при потере соединения Ktor автоматически повторяет подключение с экспоненциальным backoff (1s → 2s → 5s → 10s → 30s), до восстановления соединения котировки клиентам не поступают (WebSocket-сессии остаются открытыми, клиент получает данные как только Redis вернётся). Метрика redis_reconnect_count фиксируется в OpenTelemetry.
* Валидация и ошибки: все входные DTO валидируются до вызова сервисного слоя. Ошибки возвращаются в едином JSON-формате с errorCode, message, details, traceId.
* Наблюдаемость: подключаем OpenTelemetry SDK, автоматическую HTTP-инструментацию Ktor, ручные spans вокруг PostgreSQL transaction и Redis publish/subscribe, структурные JSON-логи.

Команда модуля:

1. Описание задачи "от заказчика"
Нам необходим единый, строгий и задокументированный стандарт обмена данными между бэкендом (Ktor) и двумя независимыми мобильными клиентами (Android Kotlin и Android Vue). Задача этого модуля — полная унификация взаимодействия. Bandz0 выступает в роли архитектора API: он должен спроектировать, задокументировать и согласовать со всеми сторонами, какие именно запросы отправляют клиенты, в какой форме (JSON и т.д.) бэкенд отдает ответы, и как транслируются потоковые данные (котировки). Это единый источник истины, чтобы при интеграции не возникало сюрпризов в виде несовпадающих полей или разных форматов ошибок.
2. Техническая реализация
(Опиши, как именно будет описан и задокументирован этот стандарт. Будет ли использоваться спецификация OpenAPI/Swagger для REST-запросов? Как будем описывать асинхронные контракты для WebSockets (например, AsyncAPI)? Где будет храниться эта документация (отдельный репозиторий, Postman-коллекция, встроенный Swagger-UI в Ktor)? Как будет реализовано версионирование API (например, /api/v1/...)?)
Модуль единого API является единым источником истины для обмена данными между Ktor-бэкендом и двумя мобильными клиентами: Android Kotlin и Android Vue. Его задача — не исполнять бизнес-логику, а строго зафиксировать контракт: какие URI существуют, какие поля и типы передаются, какие статусы и ошибки возвращаются, как выглядит поток котировок, и какие правила обратной и прямой совместимости обязаны соблюдать все команды.
Формат спецификации и документации
REST-часть API описывается в OpenAPI 3.1. Для потоковых котировок через WebSocket используется отдельная AsyncAPI-спецификация. Разделение необходимо, потому что REST и WebSocket имеют разные модели взаимодействия: для REST важны request/response-схемы и коды статуса, а для WebSocket — типы сообщений, правила подписки, формат событий и поведение при реконнекте.
OpenAPI и AsyncAPI хранятся в репозитории API-модуля в каталоге docs/api/:
   * docs/api/openapi.yaml — нормативная спецификация REST API;
   * docs/api/asyncapi.yaml — нормативная спецификация WebSocket-контракта;
   * docs/api/examples/ — примеры payload’ов для клиентов и тестов.
В Ktor-репозитории эти файлы зеркалируются как зависимость или submodule, чтобы backend не “придумывал” DTO отдельно от API-модуля. Генерация Swagger UI выполняется из openapi.yaml; UI доступен, например, по /docs/swagger, а raw-файлы — по /docs/openapi.yaml и /docs/asyncapi.yaml.
Для мобильных команд дополнительно публикуется Postman/Insomnia-коллекция, автоматически собранная из OpenAPI. Для ранней интеграции, до готовности Ktor, поднимается mock-server на основе OpenAPI/AsyncAPI в Docker Compose. Это позволяет Android Kotlin и Android Vue начать разработку экранов и DTO независимо от готовности backend-реализации.
Протоколы и сериализация
Для REST используется HTTPS + JSON UTF-8. Базовый префикс всех публичных эндпоинтов: /api/v1/. Для потоковых котировок используется WebSocket endpoint /api/v1/quotes/ws.
Во всех JSON-контрактах используется camelCase. Формат обязателен для всех участников системы: backend, Android Kotlin и Android Vue. Поля в snake_case в публичном API запрещены.
Сериализация выполняется в формате JSON. Для чисел, связанных с деньгами и количеством инструментов, используется строковое представление decimal, а не JSON number. Это критично для согласованности между Kotlin/JVM, Kotlin/Android, JavaScript/Vue и PostgreSQL numeric, чтобы не потерять точность на дробных значениях.
Нормативные правила типов:
   * UUID передается как строка, например "8e2f8d1a-1d50-4fb3-b3ea-2f88d7cbb2b1";
   * денежные значения (cash, price, amount) передаются как decimal string, например "252.0000";
   * количество бумаг (quantity) также передается как decimal string, например "10" или "10.50000000";
   * даты и время в REST передаются как ISO-8601 UTC string, например "2026-04-11T12:30:00Z";
   * время котировки в WebSocket передается как timestampMs в миллисекундах Unix Epoch;
   * перечисления передаются строками в UPPER_SNAKE_CASE только там, где это уже зафиксировано контрактом (BUY, SELL, MARKET, LIMIT, FILLED, REJECTED и т.д.).
Авторизация и безопасность
Для REST используется Authorization: Bearer <accessToken>. Access token — JWT. Refresh token не передается в заголовке и не хранится в cookie; он передается только в теле соответствующих auth-запросов (refresh, logout) и хранится сервером в PostgreSQL в виде хэша.
Для WebSocket основным способом авторизации также является Bearer token. В качестве fallback для клиентов, у которых есть ограничения на установку заголовков при WebSocket-handshake, допускается accessToken в query string. Это исключение допустимо только для /api/v1/quotes/ws; для REST query-параметр с токеном запрещен.
Версионирование
Все публичные маршруты версии 1 имеют префикс /api/v1/.... Поток котировок также версионируется тем же префиксом: /api/v1/quotes/ws.
Правила версионирования:
   * в рамках v1 допустимы только обратно совместимые изменения;
   * breaking changes выполняются только через новый префикс /api/v2/...;
   * добавление новых обязательных полей в request запрещено в рамках v1;
   * удаление, переименование или изменение типа существующего поля запрещено в рамках v1;
   * добавление новых enum-значений в v1 допускается только после согласования с лидами мобильных клиентов, потому что strict-deserialization на клиенте может не пережить “неожиданное” значение;
   * клиент обязан игнорировать неизвестные поля в response, чтобы backend мог безопасно расширять ответы в пределах v1.
Трейсинг и идентификация ответа
Каждый HTTP-ответ сервиса должен содержать X-Trace-Id. При ошибке этот же идентификатор дублируется в JSON-теле ошибки в поле traceId. Это обеспечивает сквозную трассировку между мобильным клиентом, Ktor, PostgreSQL, Redis и модулем Telemetry.
3. Контракты взаимодействия (Data Contracts)
(Это ядро твоей задачи. Распиши унифицированные форматы общения между Ktor и Мобилками:
1. Авторизация: Как передаем токены (Bearer в хедерах)?
2. REST API (CRUD): Приведи примеры структур (JSON) для основных запросов: получение портфеля, создание заявки на покупку/продажу. Точные названия полей (camelCase/snake_case) и их типы.
3. WebSockets (Котировки): Как выглядит JSON-сообщение с новой ценой, которое пушит Ktor? Как клиент подписывается на конкретный тикер?
4. Универсальный формат ошибок: Как выглядит стандартизированный ответ Ktor при ошибке (например, 400 Bad Request: {"error_code": "INSUFFICIENT_FUNDS", "message": "..."})?)
3.1. Общие правила контракта
Публичный API не использует единый “success envelope” вида { "data": ... }. Успешный ответ возвращает непосредственно предметный JSON-объект или JSON-список в согласованной обертке пагинации. Ошибка всегда возвращается в унифицированном объекте ошибки.
Content-Type для REST-запросов с телом: application/json; charset=utf-8.
Accept: application/json.
Пустой успешный ответ без тела возвращается как 204 No Content.
null имеет строгую семантику:
   * null означает “значение в данный момент отсутствует или неприменимо”;
   * null не должен подменяться пустой строкой, 0, false или пропуском поля, если это не оговорено явно;
   * для currentPrice в портфеле null означает, что актуальная цена еще не пришла из Redis/in-memory cache и клиент обязан это корректно отобразить.

3.4. WebSocket-контракт котировок
Endpoint:
GET /api/v1/quotes/ws
Авторизация:
   * основной вариант: Authorization: Bearer <accessToken>;
   * fallback: ?accessToken=<token>.
Handshake:
   * успешное подключение — HTTP 101 Switching Protocols;
   * невалидный токен — 401 UNAUTHORIZED и WebSocket не открывается.
После открытия соединения клиент сам управляет подписками. Подписка не подразумевается автоматически.
Сообщение клиента: подписка
{
 "type": "subscribe",
 "tickers": ["SBER", "GAZP"]
}
Сообщение клиента: отписка
{
 "type": "unsubscribe",
 "tickers": ["SBER"]
}
Правила:
   * type обязателен;
   * tickers обязателен для subscribe и unsubscribe;
   * тикеры передаются массивом строк;
   * пустой массив допустим, но не меняет состояние подписки;
   * дубликаты тикеров в одном сообщении должны игнорироваться сервером как технические повторы.
Сообщение сервера: новая котировка
{
 "type": "quote",
 "ticker": "SBER",
 "price": "252.0000",
 "currency": "RUB",
 "timestampMs": 1775901000000
}
Контракт:
   * сервер отправляет котировку только тем сессиям, которые подписаны на данный тикер;
   * price всегда передается как decimal string;
   * timestampMs — время тика, а не время доставки клиенту;
   * порядок сообщений гарантируется только в пределах одного соединения и по мере поступления данных из Redis;
   * гарантии доставки “каждого тика” нет: контракт WebSocket в v1 — near-real-time streaming, а не брокер сообщений с подтверждениями.
Ошибка внутри WebSocket
Для ошибок после успешного открытия сокета используется JSON того же стиля, что и HTTP, но с type: "error":
{
 "type": "error",
 "errorCode": "VALIDATION_ERROR",
 "message": "Field tickers must not be empty",
 "details": {
   "field": "tickers"
 },
 "traceId": "9b9b52d7fe2b4d45"
}
Контракт:
   * если клиент прислал корректный JSON, но с невалидной бизнес-структурой, сервер отправляет type: "error" и оставляет соединение открытым;
   * если клиент прислал невалидный JSON frame, сервер может закрыть соединение кодом протокольной ошибки;
   * heartbeat реализуется на уровне WebSocket ping/pong каждые 30 секунд, а не отдельными JSON-сообщениями.
Поведение при Redis outage и reconnect
Это критично для мобильной интеграции:
   * если Ktor теряет связь с Redis, уже открытые WebSocket-сессии не закрываются автоматически;
   * в период отсутствия Redis новые котировки клиентам не поступают;
   * после восстановления Redis поток котировок продолжается в тех же WebSocket-сессиях;
   * клиент не получает специального “catch-up dump” за время простоя;
   * если WebSocket-соединение оборвалось, клиент обязан сам переподключиться и повторно отправить свои subscribe, потому что состояние подписки связано с конкретной сессией.
3.5. Универсальный формат ошибок
Для всех REST-ошибок используется единый JSON-формат:
{
 "errorCode": "INSUFFICIENT_FUNDS",
 "message": "Not enough cash to execute order",
 "details": {
   "required": "2520.0000",
   "available": "1000.0000"
 },
 "traceId": "9b9b52d7fe2b4d45"
}
Контракт:
   * errorCode — машиночитаемый стабильный код;
   * message — короткое человекочитаемое описание;
   * details — объект с дополнительным контекстом, может быть пустым объектом {} или null, но тип должен быть согласованно задокументирован; для v1 рекомендуется всегда отдавать объект, даже пустой;
   * traceId обязателен.
Базовые errorCode:
   * VALIDATION_ERROR
   * UNAUTHORIZED
   * FORBIDDEN
   * INSTRUMENT_NOT_FOUND
   * INSUFFICIENT_FUNDS
   * INSUFFICIENT_POSITION
   * QUOTE_UNAVAILABLE
   * CONFLICT
   * INTERNAL_ERROR
Рекомендуемое соответствие HTTP-статусов:
   * 400 — VALIDATION_ERROR
   * 401 — UNAUTHORIZED
   * 403 — FORBIDDEN
   * 404 — INSTRUMENT_NOT_FOUND
   * 409 — INSUFFICIENT_FUNDS, INSUFFICIENT_POSITION, QUOTE_UNAVAILABLE, CONFLICT
   * 500 — INTERNAL_ERROR
Для мобильных клиентов нормативным считается именно errorCode, а не текст message, потому что message может меняться или локализоваться.

5. Критерии готовности (Definition of Done)
(Как мы поймем, что API спроектировано? Например: написана и опубликована полная спецификация OpenAPI, форматы обмена данными утверждены лидами Ktor (weebat) и мобилок (mryrt, Bandz0), поднят mock-сервер, REST и WebSockets контракты однозначно трактуются всеми участниками).

   1. Написана и опубликована полная спецификация OpenAPI для REST и AsyncAPI для WebSocket, и они являются официальным источником истины для Ktor и обеих мобильных команд.
   2. Для всех публичных REST endpoints версии v1 зафиксированы:

   * HTTP-методы;
   * схемы request/response;

   * примеры payload’ов;
   * коды ответа;
   * схема ошибок.
   3. Для WebSocket-контракта зафиксированы:
   * endpoint;
   * способ авторизации;
   * типы client->server сообщений;
   * типы server->client сообщений;
   * формат quote;

   * правила переподключения и повторной подписки.
   4. Явно зафиксированы межмодульные правила совместимости:

   * даты REST — ISO-8601 UTC;

1. Описание задачи "от заказчика"
Нам необходимо разработать нативное мобильное приложение под Android, которое станет основным брокерским терминалом для наших клиентов (аналог интерфейса Тинькофф Инвестиций). Пользователь должен иметь возможность авторизоваться, просматривать свой текущий портфель, искать нужные акции, следить за изменением их котировок в реальном времени (на графиках или в "стакане") и совершать сделки купли/продажи. Приложение должно быть написано на языке Kotlin с использованием современного UI-фреймворка Jetpack Compose (согласно требованиям лекции). Оно должно работать плавно, корректно обрабатывать обрывы сети и использовать единый стандарт API для связи с бэкендом (Ktor).
2. Техническая реализация
(Опиши архитектуру Android-проекта. Какой паттерн презентационного слоя используете (MVVM, MVI)? Будет ли применяться Clean Architecture (разделение на Data, Domain, UI)? Какие библиотеки берете под капот: DI (Hilt или Koin), сеть (Retrofit или Ktor Client), асинхронщина (Coroutines + StateFlow/SharedFlow), навигация (Jetpack Navigation Compose)? Будет ли локальное кэширование (Room/DataStore)?)
Нативное Android-приложение разрабатывается на Kotlin с UI на Jetpack Compose. Приложение выступает клиентом единого API-модуля и не содержит собственной бизнес-логики брокера: оно отвечает за пользовательский интерфейс, локальное состояние, сетевое взаимодействие, обработку ошибок, хранение токенов, отображение портфеля, истории операций, поиск инструментов и отправку заявок на покупку/продажу.
В качестве архитектурной основы используется Clean Architecture с разделением на слои:
      * data — работа с REST API, WebSocket, локальным кэшем, token storage;
      * domain — use case, доменные модели, интерфейсы репозиториев;
      * ui — Compose-экраны, ViewModel, ui-state, ui-events, navigation.
Для презентационного слоя используется MVVM с однонаправленным потоком данных (UDF). Это значит, что экран отправляет действия пользователя во ViewModel в виде событий, ViewModel меняет StateFlow состояния экрана, а UI только подписывается на это состояние и рендерит его. Для разовых событий — показ snackbar, переход на другой экран, открытие bottom sheet, сообщение об успешной покупке — используется отдельный поток SharedFlow/Channel.
Выбранные технологии
      * UI: Jetpack Compose, Material 3
      * Архитектура: MVVM + UDF, Clean Architecture
      * DI: Hilt
      * REST-сеть: Retrofit + OkHttp
      * Сериализация: kotlinx.serialization
      * WebSocket: OkHttp WebSocket
      * Асинхронность: Kotlin Coroutines + StateFlow/SharedFlow
      * Навигация: Navigation Compose
      * Локальное хранение токенов и настроек: DataStore
      * Локальный кэш экранных данных: Room
      * Логирование/debug: Timber
      * Мониторинг ошибок на клиенте: Crash reporting/analytics по согласованию с общей телеметрией проекта
Причины выбора
Для REST используется Retrofit + OkHttp, потому что это стабильный и хорошо поддерживаемый стек для Android, а OkHttp также закрывает WebSocket-соединение, что упрощает сетевой слой и конфигурацию перехватчиков, таймаутов и логирования.
Для JSON используется kotlinx.serialization, чтобы сохранить единый подход с Ktor-бэкендом. Это уменьшает риск расхождения по именам полей и nullable-типам.
Для dependency injection используется Hilt, так как он нативно интегрирован с Android lifecycle, ViewModel и Compose, и снижает объем шаблонного кода по сравнению с ручной DI-конфигурацией.
Архитектура модулей Android-проекта
Проект разбивается на функциональные модули или пакеты:
      * core-network — Retrofit, OkHttp, interceptors, auth handling
      * core-database — Room, DAO, кэш
      * core-model — общие DTO/мапперы/ошибки
      * feature-auth
      * feature-portfolio
      * feature-instruments
      * feature-orders
      * feature-transactions
      * feature-quotes
      * feature-profile
      * app — Application, DI graph, navigation root
Если модульность на Gradle окажется избыточной для учебного этапа, допускается старт с пакетной модульности внутри одного app-модуля, но с тем же логическим разделением слоев.
Локальное хранение и кэширование
Приложение использует два типа локального хранения:
DataStore:
      * access token
      * refresh token
      * технические флаги сессии
      * настройки пользователя, не связанные с критичными финансовыми данными
Room:
      * кэш последнего портфеля
      * кэш найденных инструментов
      * последние ордера и транзакции
      * last known quotes для экранного отображения
Кэш нужен не как источник истины для торговли, а для:
      * быстрого cold start;
      * отображения последнего известного состояния при плохой сети;
      * минимизации “прыжков” интерфейса при повторной загрузке экранов.
Источник истины для торговых операций — backend/Ktor. После успешных мутаций (buy/sell/login/refresh) приложение синхронизирует локальный кэш с ответом сервера или инициирует повторную загрузку соответствующего экрана.
Работа с денежными значениями
Это критически важное правило интеграции:
      * в API деньги и quantity приходят как строки;
      * на уровне DTO они сохраняются как String;
      * на уровне domain они преобразуются в BigDecimal;
      * UI работает уже с безопасными типами domain-уровня и форматтерами.
Использовать Double или Float для денег и количества бумаг запрещено. Это исключает расхождение с PostgreSQL numeric и серверной валидацией.
Сетевая устойчивость
Приложение должно корректно переживать:
      * временную потерю интернета;
      * истечение access token;
      * обрыв WebSocket;
      * возврат приложения из background;
      * медленный ответ сервера.
Для этого:
      * REST-клиент использует interceptor/authenticator для обновления access token через refresh endpoint;
      * WebSocket-подключение восстанавливается автоматически с backoff;
      * после переподключения клиент повторно отправляет subscribe, потому что подписка привязана к конкретной socket-session;
      * UI не очищает данные мгновенно при кратковременной потере связи, а показывает последнее кэшированное состояние и индикатор stale/offline.
Ограничение версии v1
Так как текущий унифицированный API фиксирует портфель, поиск инструментов, заявки, транзакции и поток котировок, то в v1 Android-приложение не должно обещать полноценный биржевой стакан или исторические свечи, если для этого нет отдельного backend-контракта. В рамках v1 гарантированно поддерживаются:
      * авторизация;
      * просмотр профиля;
      * просмотр портфеля;
      * поиск инструментов;
      * создание заявок buy/sell;
      * просмотр истории заявок и транзакций;
      * live-обновление котировок по подписанным тикерам.


3. Контракты взаимодействия (Data Contracts)
(Поскольку API уже унифицировано, здесь опиши внутренние контракты мобилки и интеграцию с бэкендом:
1. Сеть: Как клиент подписывается на WebSockets для получения live-котировок? Как обрабатывает потерю соединения (реконнекты)?
2. UI-контракты: Как UI (Compose-экраны) общается с ViewModel (передача интентов/ивентов и подписка на State)?
3. Модели данных: Как DTO (объекты из API) маппятся в доменные модели приложения для отображения на экране?)
3.1. Контракт сети и интеграция с backend
Android-клиент работает только через единый API-модуль и не обращается напрямую к PostgreSQL, Redis, ClickHouse или Telemetry.
Базовые правила сетевого взаимодействия:
      * REST — через /api/v1/...
      * поток котировок — через /api/v1/quotes/ws
      * все request/response используют JSON и camelCase
      * все денежные поля и количество в DTO — строки
      * даты REST — ISO-8601 UTC
      * котировки — timestampMs
      * все ошибки backend обрабатываются через errorCode, а не по тексту message
Авторизация и токены
REST
Во все защищенные запросы добавляется заголовок:
Authorization: Bearer <accessToken>
Access token хранится в DataStore и автоматически подставляется в OkHttp interceptor.
Refresh flow
Если backend возвращает 401 UNAUTHORIZED, клиент:
      1. проверяет, есть ли refresh token;
      2. выполняет POST /api/v1/auth/refresh;
      3. если refresh успешен — сохраняет новую пару токенов и повторяет исходный запрос один раз;
      4. если refresh неуспешен — очищает сессию и переводит пользователя на экран логина.
Важно:
      * одновременные параллельные 401 не должны вызывать гонку refresh-запросов;
      * обновление токена должно быть синхронизировано через mutex/single-flight механизм в auth-репозитории.
Logout
При logout клиент вызывает POST /api/v1/auth/logout, затем локально очищает DataStore, Room-кэш, socket-session и navigation back stack защищенных экранов.


3.2. Контракт WebSocket и live-котировок
WebSocket используется только для live-данных по котировкам. Endpoint:
GET /api/v1/quotes/ws
Открытие соединения
Клиент пытается авторизоваться через Bearer header. Если библиотечное или платформенное ограничение мешает передать заголовок в handshake, допускается fallback на accessToken query parameter, но внутри Android-приложения предпочтительным остается header-based вариант.
Подписка на тикеры
После открытия WebSocket клиент обязан явно отправить сообщение подписки:
{
 "type": "subscribe",
 "tickers": ["SBER", "GAZP"]
}
Для остановки обновлений:
{
 "type": "unsubscribe",
 "tickers": ["SBER"]
}
Контракт клиента:
      * клиент не предполагает автоматическую подписку на тикер после соединения;
      * список активных подписок хранится на клиенте;
      * после реконнекта клиент заново отправляет все текущие активные подписки.
Получение котировки
Сервер присылает сообщение:
{
 "type": "quote",
 "ticker": "SBER",
 "price": "252.0000",
 "currency": "RUB",
 "timestampMs": 1775901000000
}
Клиент:
      * парсит DTO;
      * преобразует price в BigDecimal;
      * обновляет in-memory quote store;
      * при необходимости обновляет Room-кэш последней известной цены;
      * пробрасывает обновление на подписанные экраны через StateFlow.
Ошибка внутри WebSocket
Если сервер прислал:
{
 "type": "error",
 "errorCode": "VALIDATION_ERROR",
 "message": "Field tickers must not be empty",
 "details": {
   "field": "tickers"
 },
 "traceId": "9b9b52d7fe2b4d45"
}
клиент:
      * не завершает сессию автоматически, если socket остается открыт;
      * отображает ошибку как непадающее UI-событие;
      * логирует traceId для диагностики.
Потеря соединения и реконнект
При обрыве WebSocket клиент переходит в состояние Disconnected и запускает автоматический reconnect с backoff, например:
      * 1 секунда
      * 2 секунды
      * 5 секунд
      * 10 секунд
      * 30 секунд
      * далее каждые 30 секунд до успеха
Во время реконнекта:
      * UI продолжает показывать последние известные цены;
      * рядом может отображаться индикатор “offline / updating paused”;
      * торговые действия через REST остаются доступны, если HTTP-сеть работает.
После успешного переподключения:
      * создается новая socket-session;
      * заново отправляются subscribe на все активные тикеры;
      * старое состояние подписок не ожидается от сервера и не восстанавливается автоматически на backend-стороне.


3.3. Внутренний UI-контракт: Compose ↔ ViewModel
UI-слой строится по принципу Intent → ViewModel → State → UI.
Каждый экран имеет:
      * UiState — полное текущее состояние экрана;
      * UiAction или UiIntent — входные действия пользователя;
      * UiEffect — одноразовые эффекты;
      * ViewModel — оркестратор use case и преобразователь domain → ui model.
Пример для экрана портфеля:
data class PortfolioUiState(
   val isLoading: Boolean = false,
   val positions: List<PositionItemUi> = emptyList(),
   val cash: String = "",
   val isOffline: Boolean = false,
   val error: String? = null
)
sealed interface PortfolioAction {
   data object Load : PortfolioAction
   data object Refresh : PortfolioAction
   data class OpenInstrument(val ticker: String) : PortfolioAction
   data class ToggleQuoteSubscription(val ticker: String, val enabled: Boolean) : PortfolioAction
}
sealed interface PortfolioEffect {
   data class ShowMessage(val message: String) : PortfolioEffect
   data class NavigateToInstrument(val ticker: String) : PortfolioEffect
}
Контракт взаимодействия:
      * Compose-экран отправляет только Action;
      * ViewModel не отдает UI сырой DTO-сетевой ответ;
      * Compose подписывается на StateFlow<UiState>;
      * одноразовые события приходят через SharedFlow<UiEffect>;
      * экран не вызывает репозитории напрямую.
Это защищает от утечек архитектуры data-слоя в UI и делает экраны тестируемыми.


3.4. Контракт доменных моделей и маппинга DTO
На уровне data используются DTO, строго повторяющие серверный JSON. На уровне domain используются внутренние модели приложения. На уровне ui — модели, оптимизированные под рендеринг.
Пример: PortfolioPosition
DTO из API:
@Serializable
data class PortfolioPositionDto(
   val ticker: String,
   val quantity: String,
   val avgPrice: String,
   val currentPrice: String?,
   val currency: String
)
Domain model:
data class PortfolioPosition(
   val ticker: String,
   val quantity: BigDecimal,
   val avgPrice: BigDecimal,
   val currentPrice: BigDecimal?,
   val currency: Currency
)
UI model:
data class PositionItemUi(
   val ticker: String,
   val quantityText: String,
   val avgPriceText: String,
   val currentPriceText: String,
   val priceChangeText: String?,
   val isPriceAvailable: Boolean
)
Правила маппинга:
      * DTO не используется напрямую в Compose;
      * String money/quantity → BigDecimal только в mapper;
      * null currentPrice в domain сохраняется как null, а в UI превращается в "—" и isPriceAvailable = false;
      * серверные enum-маппинги выполняются через безопасное преобразование с fallback на Unknown, если в будущем backend добавит значение в v1.
Пример: Order creation
Request DTO:
@Serializable
data class CreateOrderRequestDto(
   val ticker: String,
   val side: String,
   val orderType: String,
   val quantity: String,
   val limitPrice: String?
)
Domain-команда:
data class CreateOrderCommand(
   val ticker: String,
   val side: OrderSide,
   val orderType: OrderType,
   val quantity: BigDecimal,
   val limitPrice: BigDecimal?
)
Контракт:
      * ViewModel работает с CreateOrderCommand;
      * репозиторий преобразует domain-команду в DTO;
      * validation первой линии делается в UI/ViewModel, но окончательная валидация всегда на backend.
3.5. Контракт обработки ошибок
Все backend-ошибки приходят в едином формате:
{
 "errorCode": "INSUFFICIENT_FUNDS",
 "message": "Not enough cash to execute order",
 "details": {
   "required": "2520.0000",
   "available": "1000.0000"
 },
 "traceId": "9b9b52d7fe2b4d45"
}
Клиент обязан:
      * парсить errorCode;
      * маппить его в типизированную ошибку приложения;
      * отображать пользователю дружелюбное сообщение;
      * сохранять traceId в логах для диагностики.
Пример внутренней классификации:
      * UNAUTHORIZED → принудительный relogin/refresh flow
      * VALIDATION_ERROR → ошибка формы
      * INSUFFICIENT_FUNDS → ошибка торговой операции с подсказкой по балансу
      * INSUFFICIENT_POSITION → ошибка продажи
      * QUOTE_UNAVAILABLE → временная недоступность цены
      * INTERNAL_ERROR → общий fallback
Текст message с сервера не должен считаться единственным источником пользовательского текста. В Android следует иметь свою мапу user-friendly сообщений, чтобы не зависеть от изменений серверной формулировки.
3.6. Контракт экранов v1
В рамках первой версии Android-приложение обязано поддерживать:
      * экран splash/session restore;
      * экран login/register;
      * экран portfolio;
      * экран instrument search;
      * экран order placement;
      * экран orders history;
      * экран transactions history;
      * экран profile/logout.
Для каждого из этих экранов состояние делится минимум на:
      * Loading
      * Content
      * Empty
      * Error
      * Offline/Stale, если экран зависит от сети или WebSocket

3. Контракты взаимодействия (Data Contracts)
3.1. Сеть и WebSockets
REST (Axios):
         * Базовый URL задаётся через переменную окружения VITE_API_BASE_URL.

         * При получении 401 на любой запрос (кроме /auth/refresh) запускается процедура обновления токена:

3.2. WebSocket:

         * При открытии соединения в параметры URL добавляется ?accessToken=<jwt> (Capacitor WebView может иметь ограничения на кастомные заголовки в WebSocket handshake, поэтому используется fallback‑параметр).












Менеджер соединения (useWebSocket):

         * При успешном переподключении повторно отправляет все активные подписки, так как бэкенд не сохраняет их между сессиями.

Основным инструментом оркестрации для разработки и тестирования выступает Docker Compose. Для production‑окружения в будущем предусматривается переход на Kubernetes, поэтому все сервисы контейнеризируются и конфигурируются с учётом этого перехода.

/infra/docker-compose/ – основной docker-compose.yml и сопутствующие файлы:

/scripts/ – вспомогательные скрипты для управления инфраструктурой (запуск, остановка, очистка, проверка статуса).


Сеть:

3. Контракты взаимодействия (Infrastructure Contracts)
Инфраструктурные договорённости фиксируют правила, обязательные для всех команд разработки.


3.1. Сборка (Dockerfiles)


Ответственность: Dockerfile для каждого сервиса пишет команда‑разработчик этого сервиса (weebat для Ktor, girneos для Go, Bandz0 для Android Vue).


Требования от DevOps:


         * Образ должен быть основан на легковесном базовом образе (например, alpine, eclipse-temurin:17-jre-alpine для JVM, golang:1.21-alpine с последующей компиляцией в scratch).
         * Образ должен поддерживать передачу всех параметров подключения через переменные окружения.
         * Должен быть определён HEALTHCHECK (или endpoint, по которому можно настроить внешний healthcheck в compose).
         * Dockerfile располагается в корне репозитория сервиса.
         * Шаблоны: DevOps предоставляет примеры Dockerfile для каждого типа сервиса в документации.


3.2. Переменные окружения (ENV)


Единый источник конфигурации: файл .env в корне infra/docker-compose/. Он не коммитится в репозиторий, вместо этого разработчикам предоставляется .env.example с описанием всех переменных и примерами.


Переменные включают:


         * Пароли и секреты (JWT secret, пароли БД).
         * Хосты и порты для подключения (например, POSTGRES_HOST=postgres, REDIS_HOST=redis).
         * Параметры JVM, уровни логирования.


Передача в контейнеры: через environment или env_file в docker-compose.yml.


В production (Kubernetes): секреты будут передаваться через SealedSecrets или HashiCorp Vault.






3.3. Маршрутизация и порты


Внутренняя сеть: все сервисы обращаются друг к другу по имени контейнера и стандартным портам (например, jdbc:postgresql://postgres:5432/broker, redis:6379, http://ktor:8080). Эти имена зафиксированы в docker-compose.yml.


Внешние порты (на хост):


         * 8080 → Ktor (API и WebSocket)
         * 3000 → Grafana
         * 5432 → PostgreSQL (только для локальной отладки, в CI не используется)
         * 6379 → Redis (аналогично)


Безопасность: в production‑подобных конфигурациях (staging, prod) внешние порты для БД закрываются, доступ остаётся только к Ktor и Grafana (последняя через Ingress с авторизацией).


3.4. Инициализация данных


PostgreSQL: миграции выполняются самим Ktor при старте через Flyway. DevOps гарантирует, что Ktor стартует после полной готовности БД.


ClickHouse: схемы таблиц создаются скриптами, монтируемыми в /docker-entrypoint-initdb.d/. Эти скрипты готовит команда Go‑сервиса, DevOps включает их в compose.
