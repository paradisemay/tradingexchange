# CLAUDE.md — правила работы над проектом

## Архитектура
Проект — модульный монолит: один процесс, один Docker-контейнер, но код строго разделён на независимые модули (PostgreSQL-слой и Ktor бизнес-логика). Границы между модулями — только через явные интерфейсы/контракты; прямые вызовы в обход API модуля запрещены.

## Планирование и контекст
Перед началом работы над любой задачей:
1. Декомпозируй задачу на небольшие логически завершённые подзадачи
2. Запиши план в `plan.md` в корне проекта и обновляй его по ходу
3. Параллельно с кодом веди wiki в `docs/wiki/`

Структура wiki:
```
docs/
  wiki/
    modules/
      postgres/
      ktor/
```

## MCP-инструменты (использовать автоматически)
**Context7** — перед использованием любой библиотеки (Ktor, HikariCP, Flyway, kotlinx.serialization и т.д.) сначала запроси актуальную документацию через Context7, не полагайся на обучающие данные:
```
use context7 to get docs for <library>
```

**GitHub MCP** — для всех операций с репозиторием:
- Создавай PR с описанием изменений после каждого логически завершённого блока
- Открывай issues на найденные проблемы
- Связывай коммиты с issues через `fix #N` или `ref #N` в сообщении

**Snyk MCP** — запускай сканирование при:
- Добавлении новой зависимости в `build.gradle.kts`
- Перед каждым PR в main/master
- При обнаружении CVE в логах сборки
```
snyk scan project
```
Если Snyk находит критическую уязвимость (severity: high/critical) — не продолжай, зафиксируй в issue и предложи безопасную альтернативу.

## Переменные окружения — строгий запрет на хардкод
- Все токены, пароли, секреты и порты — ТОЛЬКО в переменные окружения
- При добавлении новой переменной — ОБЯЗАТЕЛЬНО обновить `.env.example`
- Хардкод credentials в коде запрещён полностью

## Коммиты (Conventional Commits)
Делай небольшие, логически завершённые коммиты:
- `feat:` — новая функциональность
- `fix:` — исправление ошибки
- `chore:` — зависимости, конфигурация
- `docs:` — документация и wiki
- `test:` — тесты
- `refactor:` — рефакторинг без изменения поведения

Один коммит = одна завершённая логическая единица. Монолитные коммиты на 500+ строк запрещены.

## Docker
- Модульный монолит разворачивается в одном контейнере, но PostgreSQL — в отдельном контейнере
- Закладывай обработку ошибок и таймауты: падение PostgreSQL не должно крашить весь процесс мгновенно
- HikariCP connection timeout, retry с экспоненциальным backoff для Redis
- Healthcheck обязателен: `/health/live` и `/health/ready`

## Обработка ошибок
При написании любого модуля сразу закладывай:
- Обработку всех ошибок — не молчи при исключениях
- Таймауты для всех внешних вызовов (JDBC, Redis, HTTP)
- Graceful degradation при недоступности зависимостей
- Единый формат ошибок: {"errorCode":"...","message":"...","details":{...},"traceId":"..."}
- Логирование с достаточным контекстом для отладки (structured JSON logs)

## Тесты
- Покрытие юнит-тестами: не менее 80%
- Интеграционные тесты через Testcontainers (PostgreSQL, Redis)
- Пиши тесты параллельно с кодом, не после
- Покрывай граничные случаи: INSUFFICIENT_FUNDS, параллельные покупки, недоступность котировок

## OpenAPI / Swagger
Параллельно с кодом генерируй и обновляй `swagger.yaml` (OpenAPI 3.0):
- Все эндпоинты документируются сразу при создании
- Обновляй спецификацию при каждом изменении API
- Храни `swagger.yaml` в корне проекта

## Финансовая логика — особые правила
- Денежные значения ТОЛЬКО в `BigDecimal`, никогда `Double`/`Float`
- Блокировки `SELECT ... FOR UPDATE` обязательны для конкурентных операций
- Порядок блокировок всегда: accounts → portfolio_positions (для исключения deadlock)
- Транзакции БД — атомарны: при любой ошибке полный rollback

## Структура кода
- Слои: routing → service → repository → database/redis
- Роуты не содержат SQL и бизнес-логики
- Один файл — одна ответственность
- Максимальная длина функции — 50 строк; если больше — декомпозируй
- Все параметры SQL передаются только через prepared statements
